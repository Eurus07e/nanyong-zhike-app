# 课表完整性、教务通知与 macOS 发布回退设计

## 目标

在不改变现有应用整体设计语言和其他业务功能的前提下，完成三项相互独立但共同进入下一版本的修复：

1. 撤销依赖 Apple Developer Program 的 Developer ID 签名、公证和 Gatekeeper 强制发布门槛，恢复无相关凭据时也能构建 macOS DMG。
2. 将整学期课表改成第 1 至第 12 节的独立纵轴，提高课程时间字符串解析兼容性，并保证任何课程都不会在统计存在时从界面静默消失。
3. 确认历次发布版教务通知失败的共同原因，保留公开接口主链路，强化数据校验和降级行为，让下一版本真正包含并验证该修复。

## 非目标

- 不实现按日期或当前教学周过滤课程。
- 不增加周次切换、今日高亮或学校校历依赖。
- 不改变课表接口的后端响应结构。
- 不重构 AI 助手、计划模块或其他读取原始课程列表的功能。
- 不尝试绕过 macOS Gatekeeper，也不声称未公证安装包可以直接双击启动。

## 已确认根因

### macOS 发布

工作区中的未提交改动新增了 Developer ID 证书导入、Apple 公证凭据检查、`notarytool`、票据装订和 Gatekeeper 校验，并让缺少凭据的 macOS 发布直接失败。维护者未加入 Apple Developer Program，因此这条发布路径不可用。

### 课表缺课

现有 `parseSchedule` 先对 `ZCXQJCDD` 执行英文逗号切分，再用单个严格正则解析。真实课程“形势与政策”的安排为：

```text
周一 7-8节 4周,8周,12周,16周 苏教D202
```

逗号切分会把离散周次拆成四段，导致所有片段均无法通过正则。课程统计直接计算原始课程数组长度，而网格只渲染解析成功的时段，于是出现“统计有课、网格没课”。“自由时间”、缺少地点、不同标点或节次写法也会被同样静默丢弃。

### 教务通知

从 `v1.1.5` 到 `v2.0.2`，已发布版本始终通过内置 `nju-cli academic-affairs notifications list` 获取列表，并把底层失败统一折叠为泛化错误。直接访问南京大学公开通知接口的主链路是在 `v2.0.2` 标签之后才进入 `main`，所以此前任何安装包都不包含这项修复。

2026-07-18 的只读验证结果如下：

- 南京大学公开通知接口可返回真实通知列表。
- 当前源码的 `PublicNoticeSource` 可读取该接口。
- 内置 `nju-cli` 的列表和 `view-html` 正文读取目前可用。

因此下一版本应保留“公开接口主链路 + 内置 CLI 降级”，并补齐真实响应形状、空数据、双链路失败和缓存回退测试，避免再次出现测试通过但发布版未覆盖真实链路的情况。

## 课表解析设计

### 输出模型

解析器不再只返回成功时段数组，而是返回一个完整结果：

- `slots`：可放入网格的已识别时段。
- `unrecognized`：无法完整放入网格的课程安排，保留课程对象和原始字符串。

每个已识别时段包含规范化后的星期、开始节次、结束节次、周次文本、地点、原始片段和课程对象。每个未识别项包含课程对象以及一个或多个未识别原始片段。

必须满足课程完整性约束：输入中的每门课程至少出现在 `slots` 或 `unrecognized` 中。若一门课程部分安排可识别、部分不可识别，则同时进入两处。

### 时段切分

不再把所有逗号视为安排分隔符。解析器扫描明确的新时段起点，只在分隔标点后出现下列标记时开始新片段：

- `周一` 至 `周日`，并兼容 `周天`。
- `星期一` 至 `星期日`，并兼容 `星期天`。
- `自由时间` 等没有固定星期和节次的安排。

这样 `4周,8周,12周,16周` 和地点说明内部的逗号会被保留，而 `...,周三 3-4节...` 会被识别为下一段安排。

### 格式兼容

节次解析至少支持：

- 单节：`3节`、`第3节`。
- 连续节次：`3-4节`、`3~4节`、`第3至4节`。
- 两节和三节连排，例如 `5-7节`、`9-11节`。
- 半角、全角和常见中文连接标点。

周次文本至少支持：

- 连续周：`1-18周`、`第1至18周`。
- 单周：`3周`。
- 离散周：`4周,8周,12周,16周` 以及 `4、8、12、16周`。
- 单双周：`2-16周(双)`、`1-17周（单）`。

地点允许缺失。星期和有效的 1 至 12 节能够确定时，缺少地点不会阻止课程进入网格，而是显示“地点待定”。无法确定固定星期或节次的安排进入未识别列表。

### 十二节网格

课表继续展示整个学期，不根据当前日期过滤。纵轴固定为：

| 节次 | 时间 |
| --- | --- |
| 第1节 | 08:00-08:50 |
| 第2节 | 09:00-09:50 |
| 第3节 | 10:10-11:00 |
| 第4节 | 11:10-12:00 |
| 第5节 | 14:00-14:50 |
| 第6节 | 15:00-15:50 |
| 第7节 | 16:10-17:00 |
| 第8节 | 17:10-18:00 |
| 第9节 | 18:30-19:20 |
| 第10节 | 19:30-20:20 |
| 第11节 | 20:30-21:20 |
| 第12节 | 21:30-22:20 |

课程块从开始节次纵向跨到结束节次。相同星期内时间重叠的课程并排布局，避免互相遮挡。课程详情继续展示完整周次、地点、教师、教学班、课程类别和学分。

### 未识别课程区域

课表下方增加与现有界面风格一致的非嵌套列表区域。每项明确显示：

```text
该课程格式暂未识别
```

并保留课程名、教学班、课程号、授课教师、学分、课程类别、开课单位和原始上课安排。用户仍可打开课程详情。该区域不是错误弹窗，也不会阻止其他课程显示。

## 教务通知设计

### 列表加载顺序

1. 优先调用南京大学本科生院公开 JSON 接口。
2. 对真实字段进行归一化，兼容 `url`、`wapUrl` 或等价链接字段，并统一日期显示。
3. 若请求失败、响应结构异常或返回数据全部无法归一化，则调用内置 `nju-cli` 作为降级来源。
4. 若两条链路均失败但进程内已有上一次成功列表，保留并返回旧列表，避免刷新操作清空可用内容。
5. 两条链路均失败且没有任何缓存时，返回明确的可重试错误。

通知详情继续使用已验证可用的 `nju-cli view-html`，保留原文链接作为详情失败时的可访问出口。链接仍限制为可信的南京大学通知域名。

### 发布覆盖

下一版本的版本说明只描述实际交付的课表与通知修复，不再描述 Apple 签名或公证。测试必须验证发布工作流没有证书/公证凭据门槛，并验证通知公开接口客户端被打入源码与发布构建路径。

## macOS 撤回范围

撤销所有为 Developer ID、公证和直接通过 Gatekeeper 而新增的内容：

- 删除发布工作流中的凭据强制检查、证书导入、正式签名、公证、票据装订、Gatekeeper 校验和临时钥匙串清理。
- PyInstaller spec 不再读取 `MACOS_CODESIGN_IDENTITY`。
- 恢复现有的临时签名 DMG 构建流程，使无 Apple 凭据的 macOS CI 可以完成。
- 删除 `docs/macos-signing.md`。
- 删除 README、使用说明、关于页面和版本说明中“已公证、可直接双击、缺少凭据阻止发布”等表述。
- 恢复首次使用 Control 点击“打开”以及必要时前往“隐私与安全性”的真实说明。
- 删除与该签名方案一同新增、仅服务于其说明的 SHA-256 发布步骤和文案。

版本号 `v2.0.3`、Windows 安装器升级以及与本任务无关的现有工作区改动不回退。

## 测试策略

### 课表

- 先为“形势与政策”的离散周次字符串编写失败测试，确认旧解析器返回 0 个时段。
- 覆盖多安排切分、单双周、中文连接符、单节、三节连排、缺少地点和自由时间。
- 验证每门输入课程都进入网格或未识别列表。
- 验证部分识别课程同时保留异常片段。
- 验证重叠时段布局结果和 1 至 12 节边界。
- 前端构建后在桌面和移动视口检查网格、跨节课程、横向滚动和未识别列表无重叠。

### 通知

- 使用 2026-07-18 实际公开接口响应形状建立固定测试夹具。
- 验证公开接口成功、公开接口失败后 CLI 降级、空/异常数据触发降级、双链路失败保留旧缓存，以及无缓存时返回错误。
- 保留通知正文清洗、可信链接和详情缓存测试。

### 发布与回归

- 静态验证工作流不包含 Developer ID、公证、`notarytool`、`stapler` 或 Apple 凭据门槛。
- 运行前端单元测试、lint、TypeScript/Vite 构建和完整 Python 测试。
- 运行 `git diff --check`，确认没有格式错误，也没有覆盖无关的用户改动。

## 风险控制

- 解析器始终保留原始字符串；未知新格式只会进入可见的未识别列表，不会再次静默消失。
- 后端课程接口不变，计划模块和 AI 助手继续消费原始课程列表。
- 通知主链路和降级链路彼此独立；失败不会清空已成功加载的通知。
- macOS 回退仅移除未具备凭据的发布能力，不更改 Windows、Linux 和桌面启动器行为。
